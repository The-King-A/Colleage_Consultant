# 智选未来 · 高考志愿助手 — 答辩亮点总结

> 适用场景：毕业设计答辩、课程设计汇报、项目路演  
> 项目名称：**Consultant（智选未来）**  
> 定位：AI + 数据驱动的**一站式**高考志愿填报辅助平台

---

## 一、一句话概括（开场 30 秒）

本系统将 **大语言模型（DeepSeek）**、**RAG 知识检索**、**可解释规则引擎** 与 **全国院校录取数据** 深度融合，覆盖考生从「认识自我 → 查院校 → 量化冲稳保 → 形成志愿方案 → AI 评审」的完整链路，并采用 **Java + Vue + Python + Node** 多技术栈微服务架构，兼顾**智能性、可复现性与工程可扩展性**。

---

## 二、六大核心亮点（建议答辩主轴线）

### 亮点 1：完整业务闭环，不是「只有一个聊天框」

| 阶段 | 功能 | 价值 |
|------|------|------|
| 认识自我 | 霍兰德兴趣测评 + AI 解读 | 从分数决策扩展到兴趣与专业匹配 |
| 量化定位 | 一分一段位次换算 | 分数转化为可比较的位次语言 |
| 数据决策 | 冲稳保测算 + 分数线洞察 | 结构化院校清单，而非纯文字建议 |
| 过程引导 | 五步填报向导（可回看/修改） | 降低填报焦虑，步骤可持久化 |
| 方案沉淀 | 志愿方案保存 + **AI 方案评审** | 从测算结果到可交付的志愿表 |
| 持续咨询 | 智能问答（RAG + 可选联网） | 政策、院校细节随时追问 |

**答辩话术**：传统志愿 App 往往只做资讯或只做推荐；本项目把**测评、测算、向导、方案、评审、问答**串成闭环，更贴近真实填报流程。

---

### 亮点 2：AI 与规则引擎「双轨并行」，互补而不替代

- **规则轨（Python）**：冲稳保按历年最低分、线差、院校省/生源省双维度匹配，结果**可复现、可解释**，适合答辩现场演示「同一输入同一输出」。
- **智能轨（Java + LangChain4j）**：志愿推荐、霍兰德解读、方案评审、智能问答，提供**自然语言解释与个性化建议**。
- **融合设计**：冲稳保页可保存方案后触发 AI 评审；院校详情「问 AI」自动带上下文跳转聊天。

**答辩话术**：纯大模型推荐存在幻觉与不可复现问题；纯规则又缺少解释力。本项目用**规则保底、AI 增值**，体现工程化思维。

---

### 亮点 3：冲稳保算法针对真实数据痛点做了工程化增强

| 问题 | 解决方案 |
|------|----------|
| 各省样本不足 | `score_synth` 按省控线 + 院校层次合成候选 |
| 样本分数扎堆、冲档为空 | 相对分档 + **均衡三等分**，保证冲/稳/保均有院校 |
| 院校省 ≠ 生源省 | 录取线按生源省匹配，院校按所在地筛选 |
| 库内无数据 | Java 端 `ensureMatcherPair` 按需补全（同省只补本地，跨省单链路，避免超时） |

**答辩话术**：算法不仅写在论文里，还处理了**样本稀疏、线差集中、跨省填报**等真实场景，体现从理论到落地的能力。

---

### 亮点 4：多技术栈分工明确，体现「会架构」而不只是「会 CRUD」

```
Vue 3 前端（5173）
    ├─ /api          → Spring Boot（8080）  主业务 + AI + 持久化
    ├─ /analytics    → FastAPI（8001）     数值分析 / 冲稳保
    └─ /report-api   → Node（3001）        HTML 报告统一模板
         ↓
    MySQL（共享） + Redis（会话 / 向量）
```

- **Java**：事务、权限、方案与向导、样本补全、LangChain4j 集成  
- **Python**：FastAPI + 规则引擎，适合快速迭代测算逻辑  
- **Node**：报告 HTML 模板集中管理，避免前端重复拼接  
- **Vue**：21 个路由、流式对话 UI、地图选校（Leaflet）

**答辩话术**：按**语言优势**拆分服务，前端统一代理，部署可独立扩缩；符合当前企业常见「核心 Java + 算法 Python + 工具 Node」的组合方式。

---

### 亮点 5：AI 能力栈完整 — RAG、流式、联网、向量记忆

| 能力 | 实现要点 |
|------|----------|
| 对话模型 | DeepSeek-V4-Flash，流式 SSE 输出 |
| 知识增强 | LangChain4j + Embedding（BGE 中文向量）+ Redis 向量库 |
| 联网检索 | Tavily，聊天页可开关，查最新招生政策 |
| 多会话 | 会话侧边栏 + memoryId 隔离上下文 |
| 垂直场景 | 志愿推荐流、霍兰德解读流、**志愿方案 AI 评审流** |

**答辩话术**：不是简单调 API，而是具备 **RAG、记忆、流式体验、工具增强（搜索）** 的完整 AI 应用形态。

---

### 亮点 6：数据规模与产品化细节到位

- **院校库**：支持全国本科院校 SQL 导入（2800+ 所量级）  
- **多维数据**：录取分数、省控线、一分一段、专业与院校属性（985/211/双一流）  
- **可视化**：数据看板、**地图选校**（城市坐标聚合）  
- **对比与导出**：院校/专业对比 + Node 统一 HTML 报告  
- **体验**：JWT 登录、收藏、个人中心、路由守卫、冲稳保/聊天全屏布局  

**答辩话术**：具备**可演示的数据厚度**和**可交付的交互细节**，不是 Demo 级空壳。

## 四、推荐答辩演示流程（约 8～10 分钟）

1. **首页**（30s）：产品定位 — AI + 数据驱动志愿辅助  
2. **智能问答**（1min）：流式回复 + 开启联网搜索，问一条招生政策类问题  
3. **冲稳保测算**（2min）：选河南+河南、输入分数 → 展示三档均有院校 → 简述分档逻辑  
4. **分数线洞察**（1min）：解释样本分布与线差，体现可解释性  
5. **位次换算**（30s）：分数 → 位次  
6. **兴趣测评**（1min）：霍兰德结果 + AI 解读  
7. **填报向导**（1min）：五步进度、展开已填步骤修改  
8. **志愿方案 + AI 评审**（1.5min）：保存冲稳保结果 → 触发评审流式输出  
9. **地图选校 / 院校对比**（1min）：数据可视化与对比导出  
10. **架构图**（1min）：四服务分工 + MySQL/Redis/DeepSeek  

> 演示前确认：MySQL 有院校数据、Redis 已启、Python 8001 在线（冲稳保）、Java 8080、前端 5173。

## 六、预期提问与参考回答（精简版）

**Q1：为什么冲稳保不用大模型直接生成？**  
A：录取匹配需要**可复现、可审计**；规则引擎基于历年分数线，同一输入结果稳定；大模型用于解读、评审和问答，职责分离。

**Q2：冲稳保「稳」档为空怎么办？**  
A：已实现绝对分档 + 相对分档 + **均衡三等分**兜底，候选≥3 所时保证冲/稳/保均有院校；样本不足时还有合成与 Java 端按需补全。

**Q3：RAG 解决什么问题？**  
A：减少模型对院校政策、填报术语的「编造」，从本地知识库检索后再生成，提高**准确性与可信度**。

**Q4：多技术栈是否增加复杂度？**  
A：有成本，但**职责清晰**：Java 管事务与 AI 集成，Python 管算法迭代，Node 管报告模板；前端统一代理，可独立部署与扩展。

**Q5：项目的实际应用价值？**  
A：降低信息不对称，把分数、位次、兴趣、院校数据转化为**可操作的志愿清单与方案**，并支持 AI 二次评审，服务考生与家长决策。

**Q6：安全性如何考虑？**  
A：JWT 鉴权、需登录功能路由守卫；生产环境应对 API Key、数据库密码做环境变量化（当前 yaml 为开发配置，答辩可说明改进方向）。
